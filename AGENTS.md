# dotfiles-installer

Monorepo for `@alexgorbatchev/dotfiles`. This root file carries shared rules; every workspace package under `packages/*` now has its own local `AGENTS.md` for package-specific guidance.

## Commands

- Install deps: `bun install --frozen-lockfile`
- Format + autofix: `bun fix`
- Lint: `bun lint`
- Typecheck: `bun typecheck`
- Full local check: `bun check`
- CI parity: `bun check:ci`
- All tests (parallel): `bun test:all`
- One test file / package: `bun test:native packages/cli/src/__tests__/generateCommand.test.ts`
- CLI against the fixture project: `bun cli --config=test-project/dotfiles.config.ts generate`
- Dashboard dev server: `bun dashboard`
- HTTP cache proxy: `bun proxy`
- Build distributable: `bun compile`
- Release build: `bun release`

## Workspace map

- Core runtime and config packages -> `packages/{arch,archive-extractor,config,core,downloader,file-system,generator-orchestrator,installer,registry,registry-database,tool-config-builder,unwrap-value,utils,version-checker}/AGENTS.md`
- Entry points, UX, and generated-output packages -> `packages/{build,cli,dashboard,e2e-test,features,http-proxy,logger,shell-emissions,shell-init-generator,shim-generator,symlink-generator,testing-helpers,virtual-env}/AGENTS.md`
- Installer plugins -> `packages/installer-{brew,cargo,curl-binary,curl-script,curl-tar,dmg,gitea,github,manual,npm,zsh-plugin}/AGENTS.md`
- Always read the nearest package-level `AGENTS.md` before editing anything under `packages/`.

## Setup and gotchas

- `AGENTS.md` is the root AI instruction file for this repository. When present, deeper `packages/*/AGENTS.md` files override it for their directories.
- Public API or behavior changes must update `.agents/skills/dotfiles/**` in the same change.
- Use `test-project/` only for live/manual CLI verification. Do not import it from automated tests. If generated output gets stale, delete `test-project/.generated/` and rerun the CLI.
- CI runs `bun install --frozen-lockfile` then `bun run check:ci`. Publish also runs `bun run compile`, and the pushed tag `vX.Y.Z` must match `package.json` version.
- When cutting a release, always create or update the matching GitHub release with `gh release` and fill in the title/body from the actual git history since the previous tag. Do not leave release details empty or rely on autogenerated placeholders.
- Bun is the only supported runtime/tooling entrypoint here. Do not use `tsc` directly.
- Keep the root `README.md` short and introductory; it should point readers to the docs website for detailed documentation.

## Shared conventions

- Implement only the requested change. Prefer the smallest complete solution and reuse existing patterns before adding new abstractions.
- Copy `defineConfig` patterns from `test-project/dotfiles.config.ts`.
- Copy `.tool.ts` patterns from `packages/e2e-test/src/__tests__/fixtures/main/tools/github-release-tool/github-release-tool.tool.ts`; copy hook usage from `packages/e2e-test/src/__tests__/fixtures/main/tools/hook-test-tool/hook-test-tool.tool.ts`.
- If any installer plugin returns `supportsSudo(): true`, update both `.sudo()` JSDoc blocks in `packages/core/src/builder/builder.types.ts` so they stay identical and list every supported installer method in the same change.

### Logging

- Logging uses the `tslog`-based safe logger wrapper; see `packages/logger/README.md` for API details.
- No `console.*` anywhere. Only CLI entrypoints and tests may create root loggers; all other code receives a logger and creates subloggers.
- Every function or method that logs must create a sublogger with `name` for structural hierarchy. Use `context` only for runtime identifiers such as tool names or hook names.
- Do not embed runtime identifiers in log messages when `context` should provide them. Keep messages short, user-facing, and translation-friendly.
- Pass error objects directly to the logger. Never extract `error.message` into log templates. User-facing mode should only surface `.tool.ts:line` locations when relevant.
- Log each failure once. Do not duplicate the same failure at multiple layers, and do not add begin/end logs around functions that already log internally.
- Do not log objects, arrays, or long string payloads. Tests that assert raw log output must match the tab separator (`\t`).
- Each package should keep one `log-messages.ts` file exporting a single `messages` object. Those files need no comments, and directly exported message objects may rely on inference.

### Testing

- Start from existing shared helpers before writing bespoke mocks or scaffolding. The main shared helpers are `FetchMockHelper`, `TestLogger`, `createMemFileSystem`, `createMockProjectConfig`, `createTestDirectories`, `createToolConfig`, `createFile`, `executeCliCommand`, and `toMatchLooseInlineSnapshot`.
- `packages/downloader/__tests__/Downloader--cached.test.ts` shows `FetchMockHelper`; `packages/cli/src/__tests__/generateCommand.test.ts` shows CLI command setup.
- `fetch` must be mocked in tests, typically with `FetchMockHelper`. When mocking a real public API, capture the fixture with `curl` first.
- Use `test-project/` only for live/manual CLI verification, never from automated tests.
- Prefer branch coverage over line coverage and keep package coverage in the 85-95% range.

### Tooling and workflow

- External data must be validated with colocated `zod` schemas. Do not use raw `fetch` outside core HTTP clients, and do not use `node:fs` outside core file system implementations.
- Use Bun's `$` shell with `.quiet()` for shell execution, and inject dependencies instead of hiding globals when possible.
- No backward-compatibility layers. When an API changes, migrate all callers, tests, and docs in the same change and delete the old path.
- Follow TDD for bug work: reproduce, add or update a failing test, fix, then run the full required checks. For CLI debugging, use `--trace --log=verbose`.
- A production change is not complete until code is formatted with `bun fix`, lint passes, typecheck passes, and the relevant full test run passes.
- If `bun install` fails with certificate errors, stop and report it instead of working around it.

## Boundaries

- Always: read the local `AGENTS.md` before changing a package that has one; run the narrowest relevant test first, then `bun check` before calling work complete.
- Ask first: public API changes, dependency additions or removals, `.github/workflows/*`, release or publish logic, package version changes, or instruction changes that affect multiple packages.
- Never: hand-edit `.dist/` or `test-project/.generated/`; add compatibility aliases/shims for old APIs; use `console.*`; bypass the HTTP/file-system abstractions with raw `fetch` or `node:fs`; switch to `tsc`; disable tests to get green CI.
