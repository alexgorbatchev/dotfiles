# @dotfiles/shim-generator

Generates executable shim scripts for installed CLI tools. Creates lightweight wrapper scripts in a bin directory that forward executions to actual tool binaries, making tools globally accessible.

## Overview

The shim-generator creates executable wrapper scripts (shims) for all installed tools. These shims are placed in a single directory that can be added to PATH, making all tools accessible system-wide without polluting PATH with multiple tool-specific directories.

## Features

- **Universal Access**: Single directory to add to PATH for all tools
- **Version Management**: Shims can redirect to different tool versions
- **Performance**: Lightweight wrappers with minimal overhead
- **Unix-like Systems**: Works on macOS, Linux, and other Unix-like systems with bash
- **Tracking**: Integrates with registry for tracking generated shims
- **Usage Telemetry**: Tracks shim execution counts in the background
- **Auto-Install**: Automatically installs tools on first use

## How Shims Work

1. Tool binary is installed under the tool directory (e.g., `{config.paths.binariesDir}/{tool}/{version-or-timestamp}/...`)
2. Installer maintains `{config.paths.binariesDir}/{tool}/current -> {version-or-timestamp}`
3. Shim executes `{config.paths.binariesDir}/{tool}/current/{binary}`
4. Shim is created at configured target directory (e.g., `{config.paths.targetDir}/{binary}`)
5. User adds target directory to PATH
6. Running `{binary}` executes the shim
7. Shim records usage asynchronously (fire-and-forget)
8. Shim forwards execution to actual binary or auto-installs if not found

Main class for generating shims.

```typescript
import { ShimGenerator } from '@dotfiles/shim-generator';

const generator = new ShimGenerator(logger, fileSystem, config, toolRegistry);
```

```typescript
interface IShimGenerator {
```

## Usage Examples

### Generate All Shims

```typescript
import { ShimGenerator } from '@dotfiles/shim-generator';

const generator = new ShimGenerator(logger, fileSystem, config, toolRegistry);

// Generate shims for all installed tools
const shimPaths = await generator.generate();

console.log('Generated shims:', shimPaths);
// [
//   '{config.paths.targetDir}/fzf',
//   '{config.paths.targetDir}/rg',
//   '{config.paths.targetDir}/bat',
// ]
```

### Generate Shims for Specific Tool

```typescript
// Generate shims only for a specific tool
const shimPaths = await generator.generateForTool('fzf');

console.log('Generated shims for fzf:', shimPaths);
// ['{config.paths.targetDir}/fzf']
```

### Integration with Installation

```typescript
import { Installer } from '@dotfiles/installer';
import { ShimGenerator } from '@dotfiles/shim-generator';

// Install tool
const installResult = await installer.install('fzf', toolConfig);

if (installResult.success) {
  // Generate shim for newly installed tool
  await shimGenerator.generateForTool('fzf');
}
```

## Generated Shim Structure

### Example Shim Script

```bash
#!/usr/bin/env bash
# Shim for fzf
# Generated by Dotfiles Management Tool

set -euo pipefail

TOOL_NAME="fzf"
BINARY_NAME="fzf"
TOOL_EXECUTABLE="{config.paths.binariesDir}/fzf/current/fzf"
GENERATOR_CLI_EXECUTABLE="path/to/cli"
CONFIG_PATH="path/to/dotfiles.config.ts"

# Check for recursion
RECURSION_ENV_VAR="DOTFILES_INSTALLING_FZF"

if [ -n "${!RECURSION_ENV_VAR:-}" ]; then
  echo "Recursive installation detected for $TOOL_NAME. Aborting to prevent infinite loop." >&2
  exit 1
fi

# Record shim usage in the background (non-blocking)
if [ "${DOTFILES_USAGE_TRACKING:-1}" != "0" ]; then
  eval "$GENERATOR_CLI_EXECUTABLE" @track-usage '"$TOOL_NAME"' '"$BINARY_NAME"' --config '"$CONFIG_PATH"' >/dev/null 2>&1 &
fi

# Check if the first argument is @update
if [ $# -gt 0 ] && [ "$1" = "@update" ]; then
  echo "Updating $TOOL_NAME to latest version..."
  eval "$GENERATOR_CLI_EXECUTABLE" update --shim-mode --config "$CONFIG_PATH" "$TOOL_NAME"
  exit $?
fi

# Check if tool exists and execute it
if [ -x "$TOOL_EXECUTABLE" ]; then
  exec "$TOOL_EXECUTABLE" "$@"
else
  # Tool not found, try to install it
  eval "$GENERATOR_CLI_EXECUTABLE" install --shim-mode --config "$CONFIG_PATH" "$TOOL_NAME"
  # If installation successful, execute binary
  if [ -x "$TOOL_EXECUTABLE" ]; then
    exec "$TOOL_EXECUTABLE" "$@"
  else
    echo "Installation completed but binary not found" >&2
    exit 1
  fi
fi
```

### Shim Characteristics

- **Executable**: Has execute permissions (755)
- **Shell Script**: Uses bash (currently bash-only, PowerShell support planned)
- **Pass-Through**: Forwards all arguments to actual binary
- **Exec**: Uses `exec` to replace process (no overhead)
- **Absolute Path**: Uses full path to binary for reliability
- **Auto-Install**: Automatically installs tool if not found
- **Update Support**: Use `{tool} @update` to update tool to latest version
- **Recursion Guard**: Prevents infinite loops if a tool calls itself during installation
- **Background Usage Tracking**: Calls private `@track-usage` command without blocking execution
- **Opt-Out Switch**: Set `DOTFILES_USAGE_TRACKING=0` to disable tracking

## Configuration

Shim paths are configured in `dotfiles.config.ts`:

```typescript
import { defineConfig } from '@alexgorbatchev/dotfiles';

export default defineConfig(() => ({
  paths: {
    targetDir: '~/.dotfiles/bin', // Where shims are created
    binariesDir: '~/.local/binaries', // Where tool binaries are located
  },
}));
```

## Best Practices

### Regenerate After Installation

```typescript
// Always regenerate shims after tool installation
await installer.install('fzf', toolConfig);
await shimGenerator.generateForTool('fzf');
```

### Validate Binary Exists

```typescript
// Check binary exists before creating shim
const binaryPath = path.join(config.paths.binariesDir, toolName, 'current', binaryName);
const exists = await fileSystem.exists(binaryPath);

if (!exists) {
  throw new Error(`Binary not found: ${binaryPath}`);
}
```

### Use Absolute Paths

```typescript
// Shims should use absolute paths
const absoluteBinaryPath = await fileSystem.realpath(binaryPath);
```

### Set Correct Permissions

```typescript
// Ensure shims are executable
await fileSystem.chmod(shimPath, 0o755);
```

## Shim vs Symlink

### Why Shims Instead of Symlinks?

**Advantages of Shims:**

- Can include initialization logic
- Can redirect to different versions
- Can set environment variables
- Can log executions
- Can validate preconditions

**Symlinks are simpler but less flexible:**

```bash
# Symlink (simple but inflexible)
ln -s ~/.dotfiles/tools/fzf/bin/fzf ~/.dotfiles/bin/fzf

# Shim (flexible and extensible)
# Can add logic before exec
```

## Advanced Usage

### Shim with Environment Variables

```bash
#!/usr/bin/env bash
# Shim with custom environment

export FZF_DEFAULT_OPTS="--height 40% --border"
exec "{config.paths.binariesDir}/fzf/current/fzf" "$@"
```

### Shim with Validation

```bash
#!/usr/bin/env bash
# Shim with precondition checking

if [[ ! -f "$HOME/.fzfrc" ]]; then
  echo "Warning: ~/.fzfrc not found" >&2
fi

exec "{config.paths.binariesDir}/fzf/current/fzf" "$@"
```

### Shim with Logging

```bash
#!/usr/bin/env bash
# Shim with execution logging

echo "[$(date)] Executing fzf with args: $*" >> ~/.dotfiles/logs/fzf.log
exec "{config.paths.binariesDir}/fzf/current/fzf" "$@"
```

## PATH Management

### Adding Shim Directory to PATH

```bash
# Add configured target directory to ~/.zshrc or ~/.bashrc
export PATH="{config.paths.targetDir}:$PATH"

# Example with actual path
export PATH="$HOME/.dotfiles/bin:$PATH"
```

### Verifying PATH

```bash
# Check shim is accessible
which fzf
# {config.paths.targetDir}/fzf

# Verify it's the shim
file $(which fzf)
# {config.paths.targetDir}/fzf: Bash script
```

## Design Decisions

### Why Bash for Shims?

Bash is used because:

- Universal availability on Unix-like systems
- Simple syntax for forwarding execution
- Good performance
- Reliable `exec` implementation
- Note: PowerShell support is planned for Windows compatibility

### Why Single Target Directory?

A single target directory (configured via `paths.targetDir`):

- Simplifies PATH management
- Makes tools discoverable
- Centralizes access control
- Eases cleanup
- User-configurable location

### Why Track in Registry?

Registry tracking enables:

- Cleanup on uninstall
- Conflict detection
- Status reporting
- Dependency management
