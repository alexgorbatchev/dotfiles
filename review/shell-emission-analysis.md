# Shell Emission Analysis: Unified Representation Opportunity

## Executive Summary

The project currently has **8 distinct types of shell content emission**, each with different:

- Configuration APIs (`.environment()`, `.aliases()`, `.functions()`, `.always()`, `.once()`, `.sourceFile()`, `.sourceFunction()`, `.completions()`)
- Storage mechanisms in `IShellInitContent`
- Formatting/wrapping strategies (subshells, functions, raw scripts)
- Execution timing (always, once, raw)

**Key Finding**: All emissions share the same fundamental pattern:

1. Content (a string of shell code or reference)
2. Execution context (HOME override, subshell isolation)
3. Timing (when to emit: startup, once, raw)
4. Optional metadata (file paths, function names, references)

This suggests an opportunity to **unify the representation** similar to how `once()`/`always()`/`raw()` work as discriminated unions for scripts.

---

## Current Emission Types

### 1. **Environment Variables** (`.environment()`)

- **API**: `.environment({ KEY: 'value' })`
- **Storage**: `IShellInitContent.environmentVariables: string[]` (as "export KEY=value")
- **Formatting**: Direct export statements
- **Timing**: Always (hoisted to top of init file)
- **HOME Override**: No
- **Example**:
  ```typescript
  shell.environment({ NODE_ENV: 'production' });
  // Emits: export NODE_ENV="production"
  ```

### 2. **Aliases** (`.aliases()`)

- **API**: `.aliases({ name: 'command' })`
- **Storage**: `IShellInitContent.toolInit: string[]` (as "alias name=command")
- **Formatting**: Direct alias statements
- **Timing**: Always
- **HOME Override**: No
- **Example**:
  ```typescript
  shell.aliases({ t: 'tool', ts: 'tool status' });
  // Emits: alias t="tool"
  ```

### 3. **Functions** (`.functions()`)

- **API**: `.functions({ funcName: 'body' })`
- **Storage**: `IShellInitContent.functions: Record<string, string>`
- **Formatting**: `FunctionScriptFormatter` - wraps in subshell with HOME override
- **Timing**: Always (emitted as function definitions)
- **HOME Override**: Yes (via subshell)
- **Example**:
  ```typescript
  shell.functions({ initTool: 'tool env --use-on-cd' });
  // Emits: initTool() { ( HOME="/home" ... ) }
  ```

### 4. **Always Scripts** (`.always()`)

- **API**: `.always('script content')`
- **Storage**: `IShellInitContent.alwaysScripts: ShellScript[]` (as `AlwaysScript` type)
- **Formatting**: `AlwaysScriptFormatter` - wraps in subshell with HOME override
- **Timing**: Every shell startup
- **HOME Override**: Yes (via subshell)
- **Example**:
  ```typescript
  shell.always('eval "$(tool init)"');
  // Emits: ( HOME="/home" ... )
  ```

### 5. **Once Scripts** (`.once()`)

- **API**: `.once('script content')`
- **Storage**: `IShellInitContent.onceScripts: ShellScript[]` (as `OnceScript` type)
- **Formatting**: `OnceScriptFormatter` - generates separate executable file
- **Timing**: Once after installation
- **HOME Override**: Yes (via subshell in generated file)
- **Example**:
  ```typescript
  shell.once('tool gen-completions');
  // Emits: separate .once/toolname-0.zsh file
  ```

### 6. **Raw Scripts** (via `.sourceFile()`, `.sourceFunction()`)

- **API**: Internal, generated by `.sourceFile()` and `.sourceFunction()`
- **Storage**: `IShellInitContent.rawScripts: string[]`
- **Formatting**: Direct emission without wrapping
- **Timing**: Always (runs at shell startup but no HOME override)
- **HOME Override**: No (but sourceFile functions have HOME in their definition)
- **Example**:
  ```typescript
  shell.sourceFunction('initFnm');
  // Emits: source <(initFnm)
  ```

### 7. **Source File** (`.sourceFile()`)

- **API**: `.sourceFile('path/to/file')`
- **Storage**: Creates **3 artifacts**:
  1. Function definition in `IShellInitContent.functions`
  2. `raw()` script for sourcing: `source <(funcName)`
  3. `raw()` script for cleanup: `unset -f funcName`
- **Formatting**: Compound - function definition + source command + cleanup
- **Timing**: Always
- **HOME Override**: Function has HOME override, source/unset commands don't
- **Example**:
  ```typescript
  shell.sourceFile('init.zsh');
  // Emits 3 artifacts:
  // 1. __dotfiles_source_mytool_0() { ( HOME="..." [[ -f "..." ]] && cat "..." ) }
  // 2. source <(__dotfiles_source_mytool_0)
  // 3. unset -f __dotfiles_source_mytool_0
  ```

### 8. **Completions** (`.completions()`)

- **API**: `.completions('path')` or `.completions({ fpath: [...] })`
- **Storage**: `IShellInitContent.completionSetup: string[]`
- **Formatting**: Shell-specific completion setup commands
- **Timing**: Always (hoisted to dedicated completions section)
- **HOME Override**: No
- **Example**:
  ```typescript
  shell.completions('completions/_tool');
  // Emits: fpath=(...) && compinit (zsh)
  ```

## Storage Structure Analysis

### Current `IShellInitContent` Interface

```typescript
export interface IShellInitContent {
  configFilePath?: string;

  // Direct script arrays
  toolInit: string[]; // aliases, misc init code
  pathModifications: string[]; // PATH exports
  environmentVariables: string[]; // export statements
  completionSetup: string[]; // completion config

  // Discriminated union scripts
  onceScripts: ShellScript[]; // OnceScript[]
  alwaysScripts: ShellScript[]; // AlwaysScript[]
  rawScripts: string[]; // plain strings

  // Special case
  functions: Record<string, string>;
}
```

### Issues with Current Structure

1. **Heterogeneous Types**: Mix of `string[]`, `ShellScript[]`, and `Record<string, string>`
2. **Unclear Relationships**: Hard to see that `sourceFile()` creates functions + raw scripts
3. **Implicit Rules**: Some content is "hoisted" (env vars, PATH), some isn't
4. **Limited Metadata**: Can't attach source information (which API call created this?)

---

## Unified Representation Proposal

### Concept: Everything is a `ShellEmission`

Similar to how `ShellScript` is a discriminated union:

```typescript
type ShellScript = OnceScript | AlwaysScript | RawScript;
```

We could represent all emissions as:

```typescript
type ShellEmission =
  | EnvironmentEmission
  | AliasEmission
  | FunctionEmission
  | ScriptEmission
  | SourceFileEmission
  | SourceFunctionEmission
  | CompletionEmission
  | PathEmission;
```

### Example Structure

```typescript
interface BaseEmission {
  kind: EmissionKind;
  toolName: string;
  configFilePath?: string;
}

interface EnvironmentEmission extends BaseEmission {
  kind: 'environment';
  variables: Record<string, string>;
  hoist: true; // Always hoisted
}

interface AliasEmission extends BaseEmission {
  kind: 'alias';
  aliases: Record<string, string>;
  hoist: false;
}

interface FunctionEmission extends BaseEmission {
  kind: 'function';
  name: string;
  body: string;
  homeOverride: true;
  hoist: false;
}

interface ScriptEmission extends BaseEmission {
  kind: 'script';
  timing: 'once' | 'always' | 'raw';
  content: string;
  homeOverride: boolean; // true for once/always, false for raw
}

interface SourceFileEmission extends BaseEmission {
  kind: 'sourceFile';
  path: string;
  // Internally creates: function + source + unset
}

interface SourceFunctionEmission extends BaseEmission {
  kind: 'sourceFunction';
  functionName: string;
  // Internally emits: source <(functionName)
}

interface CompletionEmission extends BaseEmission {
  kind: 'completion';
  config: ShellCompletionConfig;
  hoist: true; // Always hoisted to completions section
}

interface PathEmission extends BaseEmission {
  kind: 'path';
  modifications: string[];
  hoist: true;
}
```

### Benefits

1. **Uniform Interface**: All emissions have consistent metadata
2. **Explicit Rules**: Hoisting, HOME override, timing are explicit properties
3. **Traceable**: Know which API call created each emission
4. **Composable**: Easy to transform, filter, reorder emissions
5. **Type-Safe**: Single discriminated union for all emission types
6. **Extensible**: Adding new emission types is straightforward

## Current Formatters & Their Role

### Formatter Hierarchy

```
IScriptFormatter (interface)
├── AlwaysScriptFormatter      → Wraps in subshell with HOME
├── OnceScriptFormatter         → Generates separate file with self-delete
└── FunctionScriptFormatter     → Wraps function body in subshell with HOME
```

### Formatter Responsibilities

1. **HOME Override Injection**: Wrap content in subshell with `HOME=...`
2. **Shell-Specific Syntax**: Generate zsh/bash/powershell syntax
3. **Isolation**: Prevent variable pollution via subshells
4. **Cleanup**: Self-deleting scripts (once), function unset (sourceFile)

### Would Formatters Change?

With unified emissions:

- Formatters would operate on `ShellEmission` types
- Core logic remains the same (wrapping, HOME override)
- Additional formatters for new emission types (environment, alias, etc.)
- Could have a single `EmissionFormatter` with strategy pattern

## Execution Flow Comparison

### Current Flow

```
User API Call
  ↓
ShellConfigurator methods
  ↓
IShellStorage (separate fields)
  ↓
extractShellContent() → IShellInitContent
  ↓
Multiple formatters (Always, Once, Function)
  ↓
generateFileContent() → String concatenation
  ↓
Shell init file
```

### Potential Unified Flow

```
User API Call
  ↓
ShellConfigurator methods
  ↓
IShellStorage (array of ShellEmission)
  ↓
extractShellContent() → IShellInitContent
  ↓
Single EmissionFormatter (strategy per emission.kind)
  ↓
generateFileContent() → Process emissions by kind
  ↓
Shell init file
```

## Practical Examples

### Example 1: sourceFile() as Unified Emission

**Current Implementation** (3 separate operations):

```typescript
sourceFile(relativePath: string) {
  const functionName = generateName();
  const functionBody = createBody(resolvedPath);
  
  // 1. Store function
  this.storage.functions[functionName] = functionBody;
  
  // 2. Store source command
  this.storage.scripts.push(raw(sourceCommand));
  
  // 3. Store cleanup command
  this.storage.scripts.push(raw(unsetCommand));
}
```

**Unified Approach** (1 emission):

```typescript
sourceFile(relativePath: string) {
  this.storage.emissions.push({
    kind: 'sourceFile',
    path: resolvedPath,
    toolName: this.toolName,
    // Formatter handles function def + source + unset
  });
}
```

### Example 2: Querying Emissions

**With Unified Structure**:

```typescript
// Find all emissions that need HOME override
const needsHome = emissions.filter((e) =>
  e.kind === 'function' ||
  e.kind === 'script' && e.timing !== 'raw'
);

// Find all hoisted content
const hoisted = emissions.filter((e) => e.hoist === true);

// Reorder emissions by priority
const ordered = emissions.sort((a, b) => EMISSION_PRIORITY[a.kind] - EMISSION_PRIORITY[b.kind]);
```

**Current Structure**: Need separate handling for each field

## Migration Considerations

### Would This Be Backward Compatible?

- **User-facing API**: No changes needed
- **Internal API**: `IShellInitContent` structure would change
- **Formatters**: Would need refactoring but core logic same
- **Tests**: Would need updates

### Migration Path

1. Introduce `ShellEmission` types alongside current structure
2. Emit both old and new structures during transition
3. Update formatters to work with emissions
4. Update generators to use emissions
5. Remove old structure

### Complexity Assessment

- **High Initial Cost**: Significant refactoring
- **Long-term Benefit**: Much easier to add new emission types
- **Maintenance**: Clearer, more maintainable codebase

## Emission Order (Current Implementation)

The generated shell init file follows a **fixed order** defined in `BaseShellGenerator.generateFileContent()`:

### Global Emission Order

```
1. File Header (metadata, comments)
2. Once Script Initialization (if any tools use once scripts)
3. PATH Modifications (HOISTED from all tools)
4. Environment Variables (HOISTED from all tools)
5. Tool-Specific Initializations (per-tool, in iteration order)
6. Shell Completions Setup (HOISTED from all tools)
7. End of File marker
```

### Per-Tool Emission Order

Within the "Tool-Specific Initializations" section, each tool emits in this order:

```
1. Tool Header (config file path comment)
2. Functions (via .functions())
3. Always Scripts (via .always())
4. Raw Scripts (via .sourceFile(), .sourceFunction())
5. Tool Init (aliases via .aliases())
```

### Order Implications

| Emission Type | Position                  | Why                                                      |
| ------------- | ------------------------- | -------------------------------------------------------- |
| PATH          | Global (hoisted)          | Must be available before any tool code runs              |
| Environment   | Global (hoisted)          | Variables needed by tool init code                       |
| Functions     | Per-tool, first           | Must be defined before `.sourceFunction()` uses them     |
| Always        | Per-tool, after functions | Can call functions defined above                         |
| Raw           | Per-tool, after always    | `source <(funcName)` must come after function definition |
| Aliases       | Per-tool, last            | Simple, no dependencies                                  |
| Completions   | Global, at end            | Only needed for interactive sessions                     |

### Once Scripts: Special Case

Once scripts are **not emitted inline**. Instead:

1. `OnceScriptInitializer` adds a loop that scans `.once/` directory
2. `OnceScriptFormatter` generates separate executable files in `.once/`
3. At shell startup, each `.once/*.zsh` file is sourced then self-deletes

### Tool Iteration Order

Tools are emitted in **Map iteration order** (insertion order):

```typescript
for (const [toolName, content] of toolContents) {
  // ...
}
```

This is determined by the order `extractToolContents()` processes tool configs.

### Order-Related Bugs/Issues

1. **No explicit dependency resolution**: Tool A can't declare "emit after Tool B"
2. **No priority system**: All tools have equal weight
3. **Function cross-references**: If Tool A's `.sourceFunction()` references Tool B's function, order matters

### Would Unified Emissions Help With Order?

Yes! A unified structure could include:

```typescript
interface ShellEmission {
  kind: EmissionKind;
  priority?: number; // Explicit ordering
  phase: 'hoisted' | 'tool' | 'completion'; // Where to emit
  dependencies?: string[]; // Emit after these emissions
}
```

This would enable:

- **Explicit priority**: `{ priority: 100 }` for critical PATH setup
- **Dependency declaration**: "Emit this after function X is defined"
- **Phase targeting**: Choose hoisted vs per-tool placement

## Questions for Decision Making

1. **Is the abstraction worth it?**
   - How often do we add new emission types?
   - How often do we need to query/transform emissions?

2. **What immediate benefits?**
   - Better traceability (know where emissions came from)
   - Easier debugging
   - More consistent error handling

3. **Performance implications?**
   - Array of objects vs separate arrays
   - Likely negligible for typical use cases

4. **Testing impact?**
   - Would need to rewrite many tests
   - But tests might become simpler with unified structure

## Recommendation

### Short Term

**Keep current structure** if:

- No immediate need to add new emission types
- Current system is working well
- Team bandwidth is limited

**Consider unification** if:

- Planning to add more emission types (e.g., completion modes, PATH manipulation strategies)
- Want better tooling for emission analysis
- Experiencing bugs related to emission ordering or interference

### Long Term

A unified `ShellEmission` type provides better:

- **Maintainability**: Clear, consistent structure
- **Extensibility**: Easy to add new emission types
- **Debuggability**: Complete metadata on every emission
- **Testability**: Uniform testing patterns

The pattern already exists in the codebase (`ShellScript` discriminated union), so extending it to all emissions would be consistent with existing design.

---

## Alternative: Partial Unification

Instead of full unification, consider **just unifying the "output strategy"**:

```typescript
interface EmissionStrategy {
  timing: 'once' | 'always' | 'raw';
  homeOverride: boolean;
  hoist: boolean;
  formatter: (content: string, shellType: ShellType) => string;
}

// Then each emission type declares its strategy
const ALIAS_STRATEGY: EmissionStrategy = {
  timing: 'always',
  homeOverride: false,
  hoist: false,
  formatter: (content) => content, // direct emission
};
```

This would:

- Keep separate storage arrays
- Unify how content is formatted/emitted
- Provide clear documentation of emission behavior
- Lower migration cost

## Blocks: Structured Output Organization

### Concept

Instead of ad-hoc string concatenation, emissions are organized into **Blocks** - structural units that represent sections of the output file.

```typescript
interface IBlock {
  id: string; // Unique identifier
  kind: BlockKind; // header, section, tool, footer
  title?: string; // Human-readable title
  priority: number; // Sort order (lower = earlier)
  emissions: ShellEmission[]; // Content within this block
  children?: IBlock[]; // Nested blocks (e.g., tool within section)
  metadata?: BlockMetadata; // Attribution, source file, etc.
}

type BlockKind =
  | 'file-header' // Generated file header
  | 'section' // Major section (PATH, Environment, etc.)
  | 'tool' // Tool-specific content
  | 'file-footer'; // End of file marker
```

### Block Hierarchy

```
FileBlock (kind: 'file-header')
├── SectionBlock: "Once Script Initialization" (priority: 100)
├── SectionBlock: "PATH Modifications" (priority: 200)
│   └── emissions: [PathEmission, PathEmission, ...]
├── SectionBlock: "Environment Variables" (priority: 300)
│   └── emissions: [EnvironmentEmission, ...]
├── SectionBlock: "Tool-Specific Initializations" (priority: 400)
│   ├── ToolBlock: "fnm" (priority: 1)
│   │   └── emissions: [FunctionEmission, ScriptEmission, ...]
│   ├── ToolBlock: "zoxide" (priority: 2)
│   │   └── emissions: [FunctionEmission, ...]
│   └── ...
├── SectionBlock: "Shell Completions" (priority: 500)
│   └── emissions: [CompletionEmission, ...]
└── FileBlock (kind: 'file-footer')
```

### Block Builder API

```typescript
const builder = new BlockBuilder();

// Build file structure
const file = builder
  .fileHeader({ dotfilesDir: '/path/to/dotfiles' })
  .section('once-init', { title: 'Once Script Initialization', priority: 100 })
  .section('path', { title: 'PATH Modifications', priority: 200, hoist: true })
  .section('env', { title: 'Environment Variables', priority: 300, hoist: true })
  .section('tools', { title: 'Tool-Specific Initializations', priority: 400 })
  .section('completions', { title: 'Shell Completions', priority: 500, hoist: true })
  .fileFooter()
  .build();

// Add tool blocks within the tools section
file.getSection('tools')
  .addToolBlock('fnm', { configFilePath: '/path/to/fnm.tool.ts' })
  .addToolBlock('zoxide', { configFilePath: '/path/to/zoxide.tool.ts' });

// Add emissions to blocks
file.getSection('path').addEmission(pathEmission);
file.getSection('tools').getTool('fnm').addEmission(functionEmission);
```

### Block Metadata

```typescript
interface BlockMetadata {
  // Attribution
  sourceFile?: string; // Config file that generated this
  sourceTools?: string[]; // For hoisted content

  // Documentation
  description?: string; // What this block does
  hoistingExplanation?: string; // Why content was hoisted here

  // Generation info
  generatedAt?: Date;
  generatorVersion?: string;
}
```

## Platform Formatters

Each shell type has its own formatter that knows how to render emissions and blocks.

### Formatter Interface

```typescript
interface IEmissionFormatter {
  readonly shellType: ShellType;
  readonly fileExtension: string;

  // Emission formatting
  formatEmission(emission: ShellEmission): string;
  formatEnvironment(emission: EnvironmentEmission): string;
  formatAlias(emission: AliasEmission): string;
  formatFunction(emission: FunctionEmission): string;
  formatScript(emission: ScriptEmission): string;
  formatCompletion(emission: CompletionEmission): string;
  formatPath(emission: PathEmission): string;

  // Block rendering
  renderBlock(block: IBlock): string;
  renderFileHeader(metadata: BlockMetadata): string;
  renderSectionHeader(title: string): string;
  renderToolHeader(toolName: string, metadata?: BlockMetadata): string;
  renderFileFooter(): string;

  // Comments
  commentLine(text: string): string;
  headerLine(char: string, title?: string): string;
}
```

### Example: ZshFormatter

```typescript
class ZshFormatter implements IEmissionFormatter {
  readonly shellType = 'zsh' as const;
  readonly fileExtension = '.zsh';

  formatEnvironment(emission: EnvironmentEmission): string {
    return Object.entries(emission.variables)
      .map(([key, value]) => `export ${key}="${value}"`)
      .join('\n');
  }

  formatFunction(emission: FunctionEmission): string {
    // Wrap in subshell with HOME override
    return dedent`
      ${emission.name}() {
        (
          HOME="${this.homeDir}"
          ${emission.body}
        )
      }
    `;
  }

  formatScript(emission: ScriptEmission): string {
    if (emission.timing === 'raw') {
      return emission.content;
    }

    // Wrap in subshell with HOME override
    return dedent`
      (
        HOME="${this.homeDir}"
        ${emission.content}
      )
    `;
  }

  renderSectionHeader(title: string): string {
    return this.headerLine('=', title);
  }

  commentLine(text: string): string {
    return `# ${text}`;
  }

  headerLine(char: string, title?: string): string {
    const width = 80;
    if (!title) return `# ${char.repeat(width - 2)}`;
    const padding = char.repeat((width - title.length - 4) / 2);
    return `# ${padding} ${title} ${padding}`;
  }
}
```

### Formatter Factory

```typescript
class EmissionFormatterFactory {
  static create(shellType: ShellType, config: FormatterConfig): IEmissionFormatter {
    switch (shellType) {
      case 'zsh':
        return new ZshFormatter(config);
      case 'bash':
        return new BashFormatter(config);
      case 'powershell':
        return new PowerShellFormatter(config);
      default:
        throw new Error(`Unsupported shell type: ${shellType}`);
    }
  }
}
```

## Complete Rendering Pipeline

```typescript
// 1. Collect emissions from tool configs
const emissions: ShellEmission[] = toolConfigs.flatMap(extractEmissions);

// 2. Build block structure
const blockBuilder = new BlockBuilder(defaultBlockStructure);
const blocks = blockBuilder.build(emissions);

// 3. Sort/reorder blocks and emissions
const sorter = new BlockSorter();
const sortedBlocks = sorter.sort(blocks);

// 4. Render with platform formatter
const formatter = EmissionFormatterFactory.create('zsh', config);
const renderer = new ShellBlockRenderer(formatter);
const output = renderer.render(sortedBlocks);

// 5. Write to file
await fs.writeFile(outputPath, output);
```

### Block Renderer

```typescript
class ShellBlockRenderer implements IBlockRenderer {
  constructor(private formatter: IEmissionFormatter) {}

  render(blocks: IBlock[]): string {
    return blocks.map((block) => this.renderBlock(block)).join('\n');
  }

  private renderBlock(block: IBlock): string {
    const parts: string[] = [];

    // Render block header
    switch (block.kind) {
      case 'file-header':
        parts.push(this.formatter.renderFileHeader(block.metadata));
        break;
      case 'section':
        parts.push(this.formatter.renderSectionHeader(block.title!));
        if (block.metadata?.hoistingExplanation) {
          parts.push(this.formatter.commentLine(block.metadata.hoistingExplanation));
        }
        break;
      case 'tool':
        parts.push(this.formatter.renderToolHeader(block.title!, block.metadata));
        break;
      case 'file-footer':
        parts.push(this.formatter.renderFileFooter());
        break;
    }

    // Render emissions in this block
    for (const emission of block.emissions) {
      if (block.metadata?.sourceTools?.length) {
        parts.push(this.formatter.commentLine(`Hoisted from: ${block.metadata.sourceTools.join(', ')}`));
      }
      parts.push(this.formatter.formatEmission(emission));
    }

    // Render child blocks
    if (block.children) {
      for (const child of block.children) {
        parts.push(this.renderBlock(child));
      }
    }

    return parts.join('\n');
  }
}
```

## Benefits of This Architecture

### 1. **Complete Separation of Concerns**

| Component  | Responsibility                 |
| ---------- | ------------------------------ |
| Emissions  | What to emit (data)            |
| Blocks     | Where to emit (structure)      |
| Formatters | How to emit (shell syntax)     |
| Renderers  | Assembly (putting it together) |

### 2. **Platform Independence**

Emissions are shell-agnostic:

```typescript
// Same emission, different output
const envEmission: EnvironmentEmission = {
  kind: 'environment',
  variables: { NODE_ENV: 'production' },
};

zshFormatter.formatEnvironment(envEmission);
// → export NODE_ENV="production"

powershellFormatter.formatEnvironment(envEmission);
// → $env:NODE_ENV = "production"
```

### 3. **Structured Comments/Headers**

Blocks provide consistent, structured output:

```zsh
# ==============================================================================
```

### 4. **Testability**

Each component is independently testable:

### 5. **Extensibility**

Adding new emission types is isolated:

```typescript
// 1. Define emission type
interface ConditionEmission extends BaseEmission {
  kind: 'condition';
  condition: string;
  ifTrue: ShellEmission[];
  ifFalse?: ShellEmission[];
}

// 2. Add formatter methods
formatCondition(emission: ConditionEmission): string {
  return `if ${emission.condition}; then\n${...}\nfi`;
}

// No changes to blocks, renderers, or other emissions
```

## Example: Full Pipeline

### Input (Tool Config)

```typescript
defineTool((install, ctx) =>
  install('github-release', { repo: 'Schniz/fnm' })
    .bin('fnm')
    .zsh((shell) =>
      shell
        .environment({ FNM_DIR: ctx.currentDir })
        .functions({ initFnm: 'fnm env --use-on-cd' })
        .sourceFunction('initFnm')
    )
);
```

### Emissions Generated

```typescript
[
  { kind: 'environment', variables: { FNM_DIR: '/path' }, hoist: true },
  { kind: 'function', name: 'initFnm', body: 'fnm env --use-on-cd', homeOverride: true },
  { kind: 'sourceFunction', functionName: 'initFnm' },
];
```
