# System Patterns

This document outlines the system architecture, key technical decisions, and design patterns used within this dotfiles project, now centered around a TypeScript/Bun management tool.

## System Architecture

The project is transitioning from a collection of shell scripts to a structured system managed by a central TypeScript/Bun application. This application reads tool configurations and generates the necessary components for the environment setup: Bash shims, a consolidated Zsh initialization file, and symbolic links for configuration files.

## Key Technical Decisions

- **Version Control:** Git remains the primary tool for tracking changes to the dotfiles repository, including the new TypeScript configuration files and the management tool's source code.
- **TypeScript/Bun Management Tool:** A core decision is to build a dedicated CLI tool in TypeScript using Bun for its speed and modern features. This tool orchestrates the entire dotfiles management process.
- **No Embedding of Other Managers:** The tool will *not* embed or wrap other package managers (like Homebrew, etc.). Instead, it will generate scripts or configurations that call out to these managers if they are available on the user's system. This keeps the tool lightweight and avoids tightly coupling it to other systems.
- **Generic Bash Shims:** Instead of Zsh aliases, generic Bash shims will be generated in a global PATH location (configurable, defaulting to `/usr/bin`) to provide universal access to tools. These shims check for the binary's existence and either execute it or trigger installation.
- **Location-Independent Path Resolution:** The ShimGenerator uses ES modules' `import.meta.url` feature to determine paths, making it location-independent and not reliant on `appConfig.dotfilesDir`. This ensures the generator works correctly regardless of its installation location.
- **Centralized TypeScript Installation:** A TypeScript installation script handles all installation logic, rather than embedding it in the shims. This script is called by the shim when the binary is not found. The installation script (`install-tool.sh`) is a Bash wrapper that invokes the main CLI's installation command.
- **Generated Zsh Init:** A single Zsh initialization file will be generated by the management tool, consolidating all shell-specific configurations and replacing the need to source individual tool `init.zsh` files.
- **Symbolic Links:** Still used for linking configuration files from the repository to their expected locations in the home directory, orchestrated by the management tool.
- **File System Abstraction & Dependency Injection (DI):**
    - The project uses a file system abstraction (`IFileSystem`) to support both real (`NodeFileSystem`) and virtual (`MemFileSystem`) file systems.
    - **Command-Specific Service Instantiation:** Command action handlers (e.g., in `generateCommand.ts`, `installCommand.ts`) are now responsible for calling `setupServices` (from [`src/cli.ts`](src/cli.ts:1)) to obtain fresh instances of core services like `AppConfig`, `IFileSystem`, `IGeneratorOrchestrator`, `IInstaller`, etc. This ensures services are configured appropriately for each command's specific context (e.g., respecting a `--dry-run` option passed to that command).
    - **Simplified Command Registration:** As a consequence of services being instantiated within action handlers, most `register...Command` functions (e.g., `registerGenerateCommand`) have been simplified. They primarily accept only the Commander `program` instance and no longer take core services as direct parameters.
    - **`cleanupCommand` DI Exception:** The `registerCleanupCommand` function in [`cli.ts`](src/cli.ts:1) represents an exception. It still passes `AppConfig`, `IFileSystem`, and a `clientLogger` (created in `cli.ts` specifically for this command's registration) to the `registerCleanupCommand` function in [`src/modules/cli/cleanupCommand.ts`](src/modules/cli/cleanupCommand.ts:1). This is because the `CleanupCommand` class itself takes these dependencies (including the logger) in its constructor, rather than its action handler creating them.
- **Delayed Installation via Shims:** Installation is triggered only on the first execution of the tool, but the complex installation logic is handled by the TypeScript script rather than the shim itself.
- **Configuration Management:** A `.env` file with sensible defaults is used for configuration, with a centralized configuration module that allows overriding via command-line arguments. This includes making the directory for tool configuration files (`TOOL_CONFIG_DIR`) specifiable.
- **GitHub API Integration:** A dedicated GitHub API client handles fetching release information.
- **System Command Execution:** The `zx` library is used within the TypeScript installation script to reliably execute system commands (e.g., `tar`, `unzip`, `gunzip`) for tasks like archive extraction.
- **Modular Type Definitions and Documentation:** Project-wide shared types are refactored from a single [`src/types.ts`](src/types.ts:1) file into a dedicated `src/types/` directory. Each type or group of related types resides in its own file (e.g., [`appConfig.types.ts`](src/types/appConfig.types.ts:0), [`archive.types.ts`](src/types/archive.types.ts:0)), and all are re-exported via a barrel file ([`src/types/index.ts`](src/types/index.ts:1)). This modularization improves manageability and discoverability.
  - **Comprehensive JSDoc:** All exported types, interfaces, and their properties within these new type files are documented with comprehensive JSDoc comments. This includes detailed explanations, usage examples (especially for callbacks), and clarifications for installation method parameters. This practice significantly enhances code clarity, maintainability, and developer understanding of the type system.
- **Standardized CLI Logging with `clientLogger`:**
    - The CLI employs a `clientLogger` (from `src/modules/logger/createClientLogger.ts`, based on `consola`) for all user-facing output, replacing direct `console.*` calls (e.g., `console.log`, `console.warn`).
    - **Command-Specific Logger Instantiation:** Each command's action handler is now responsible for creating its own `clientLogger` instance by calling `createClientLogger` (from `@modules/logger/clientLogger`). This logger is configured based on command-line options like `--verbose` and `--quiet` passed to that specific command.
    - This ensures consistent output that respects verbosity flags and the `NODE_ENV=test` environment variable (which silences the logger).
    - Standard messages use `logger.info()`, detailed debug messages use `logger.debug()` (enabled by `--verbose`), and warnings can use `logger.warn()`.
    - An exception is a deliberate bootstrap `console.error` in `cli.ts` for critical early errors before the logger is initialized.
    - This standardization improves user experience by providing predictable and controllable CLI output, and enhances maintainability by centralizing logging logic.
- **Commander.js Test Isolation Strategy:**
- To prevent state leakage and command re-registration errors when testing Commander.js applications, the main `Command` instance (the `program` object) is instantiated *within* the primary CLI execution function (e.g., `main()` in [`src/cli.ts`](src/cli.ts:1)).
- This ensures that each test run invoking the main execution function receives a fresh, isolated `program` instance, crucial for reliable and independent test execution.

## Design Patterns

- **Centralized Configuration:** Tool configurations are centralized in structured TypeScript files.
- **Code Generation:** The management tool uses code generation to create shims and the Zsh init file.
- **Idempotency:** The generation and cleanup processes should be idempotent.
- **Modular Tool Configuration:** Each tool's configuration is defined in a separate TypeScript file (`*.tool.ts`). These files are dynamically loaded and validated (using `ToolConfigSchema` via the `loadToolConfigs` function) by the CLI to provide the necessary `ToolConfig` objects for the system. This pattern centralizes how user-defined tool specifications are processed as system inputs.
- **Discriminated Union for `ToolConfig`:** The `ToolConfig` type has been refactored into a discriminated union based on the `installationMethod` property. This ensures that the `installParams` object only contains properties relevant to the chosen installation method, significantly improving type safety and configuration validation. For tools that do not have a direct installation process managed by the installer (e.g., built-in shell commands or tools managed externally), `installationMethod: 'none'` is used, and `installParams` is typically empty or omitted. This change is reflected in the `ToolConfigSchema` for validation and impacts how the `ToolConfigBuilder` constructs `ToolConfig` objects.
- **Command Pattern:** The management tool provides distinct commands (`generate`, `cleanup`, etc.) for different operations. Each command's action handler is responsible for setting up its required services and logger.
- **Dependency Injection (Command-Scoped):** Services are injected at the command action level by calling `setupServices`. This provides command-specific instances.
- **Environment-based Configuration:** The `.env` file and centralized config module provide a flexible configuration system.
- **Repository Pattern:** The GitHub API client encapsulates all interactions with the GitHub API, providing a clean interface for fetching release information.
- **System Command Execution (via zx):** Using `zx` simplifies running and managing shell commands within the TypeScript code.

## Component Relationships

- The dotfiles repository contains the TypeScript management tool source code (organized into modules under `src/modules/`) and the tool configuration files (typically located within `src/tools/`, and configurable via `TOOL_CONFIG_DIR` as defined in `.env`).
- The management tool reads the TypeScript configurations from the specified/default directory.
- The management tool generates generic Bash shims into a target directory (e.g., `/usr/bin`).
- The management tool generates a single Zsh init file.
- The management tool creates symbolic links for configuration files.
- The user's main Zsh profile sources the generated Zsh init file.
- The generated Bash shims check for binary existence and either:
  - Execute the binary directly if it exists
  -   Call the installation script (`install-tool.sh`) if the binary doesn't exist, which in turn invokes the main CLI's installation command
  - The TypeScript installation script (orchestrated by the `Installer` module in [`src/modules/installer/Installer.ts`](src/modules/installer/Installer.ts:0)) handles all installation logic. This includes:
      - Downloading files.
      - Extracting archives (using system tools via `zx`).
      - Placing binaries in `.dotfiles/.generated/binaries/<tool_name>/<version>/<binary_name>`.
      - Creating a symlink in `.dotfiles/.generated/bin/<tool_name>` pointing to the actual binary. The `Installer` records this `symlinkPath` in the `InstallResult`.
      - Caching downloaded files in `.dotfiles/.generated/cache/<tool_name>/`.
      - Recording any other significant actions or messages in the `otherChanges` field of the `InstallResult`. These messages include details about downloads, extractions, temporary file handling, permission changes, and binary placement, providing a comprehensive log of the installation process.
  - The GitHub API client fetches release information from GitHub for tools installed from GitHub releases.
  - A manifest file (`GeneratedArtifactsManifest` in `types.ts`, typically stored at `.generated/manifest.json`) tracks generated artifacts, including shim paths, shell initialization file path, detailed symlink operation results, and the last generation timestamp.
  - The `install` command in the CLI ([`src/cli.ts`](src/cli.ts:1)) consumes the `InstallResult` from the `Installer`. It displays the `symlinkPath` and iterates through the `otherChanges` array, printing each message to provide the user with a detailed, step-by-step account of the installation.
  
  ### Key Generator Modules

-   **`generator-shim`**: This module is responsible for generating executable Bash shims for tools.
    *   **Core Logic**: It takes `ToolConfig` data and generates shim files that, when executed, will either run the already installed tool or trigger its installation via the main CLI.
    *   **Path Resolution**: Uses ES modules' `import.meta.url` feature to determine the absolute path to the installation script, making it location-independent and not reliant on `appConfig.dotfilesDir`.
    *   **Interface/Return**: Implements `IShimGenerator`. Its `generate()` method now returns `Promise<string[]>` detailing the paths of the generated shim files. It operates on the `IFileSystem` instance injected by the CLI, performing real or in-memory operations accordingly.
    *   **Output**: Bash script files placed in the configured shim directory (e.g., `AppConfig.targetDir`).

-   **`generator-shell-init`**: This module is responsible for creating a consolidated Zsh initialization file.
    *   **Core Logic**: It processes `ToolConfig` entries (specifically the `zsh()` and `completions()` configurations) from all defined tools to build the final Zsh init script. This includes aggregating `PATH` modifications, environment variables, `zshInit` snippets, and setting up Zsh completion.
    *   **Interface/Return**: Implements `IShellInitGenerator`. Its `generate()` method now returns `Promise<string | null>`, providing the path to the generated shell initialization file, or `null` if no file was generated. It operates on the `IFileSystem` instance injected by the CLI.
    *   **Output Characteristics**: The generated Zsh init file is designed to be *sourced* by the user's main Zsh configuration file.

-   **`generator-symlink`**: This module is responsible for creating symbolic links for dotfiles based on `ToolConfig`.
    *   **Core Logic**: It processes the `symlinks` array from `ToolConfig`, resolving source and target paths, handling existing items based on `overwrite` and `backup` options, and creating the necessary symlinks.
    *   **Home Directory Resolution**: Uses `appConfig.homeDir` directly for path resolution, ensuring symlinks work correctly in isolated environments like E2E tests.
    *   **Interface/Return**: Implements `ISymlinkGenerator`. Its `generate()` method now returns `Promise<SymlinkOperationResult[]>`, providing detailed outcomes for each symlink operation (e.g., created, skipped, backed_up_and_created). It operates on the `IFileSystem` instance injected by the CLI.
    *   **Test Compliance**: The module's tests (`SymlinkGenerator.test.ts`) are fully compliant with project testing rules.

-   **`generator-orchestrator`**: This module coordinates the artifact generation process.
    *   **Core Logic**: It utilizes the `generator-shim`, `generator-shell-init`, and `generator-symlink` modules to generate all necessary artifacts. It receives an `IFileSystem` instance from the CLI and passes it down to the individual generator modules, thus determining whether operations are performed on the real file system or in memory.
    *   **Manifest Management**: It is responsible for reading the existing `GeneratedArtifactsManifest` (if any), invoking the individual generator modules, consuming their detailed return values (shim paths, shell init path, symlink results), and then updating and writing the `GeneratedArtifactsManifest` with this comprehensive information. This ensures an accurate record of all generated files and their statuses.
    *   **Interface**: Implements `IGeneratorOrchestrator`, typically with a `generateArtifacts()` method.

### CLI (Command Line Interface)

-   **Location:** [`src/cli.ts`](src/cli.ts:1)
-   **Library:** `commander` is used for parsing command-line arguments and defining commands.
-   **Dependency Injection (DI) and Service Setup:**
    *   The `setupServices` function in [`cli.ts`](src/cli.ts:1) is responsible for instantiating and wiring up core services (e.g., `AppConfig`, `IFileSystem`, `IGeneratorOrchestrator`, `IInstaller`).
    *   Command action handlers (e.g., in `generateCommand.ts`) call `setupServices` to obtain these services, configured for the command's context (e.g., `--dry-run`).
    *   This approach centralizes service creation while allowing command-specific configurations.
-   **Client Logger Instantiation:**
    *   Each command's action handler calls `createClientLogger` (from `@modules/logger`) to create its own `clientLogger` instance.
    *   This logger is configured based on command-line options like `--verbose` and `--quiet` specific to that command.
-   **`generate` Command:**
    *   **Purpose:** The `generate` command triggers the `GeneratorOrchestrator` (obtained via `setupServices`) to generate all defined artifacts.
    *   **Options:** It accepts a `--dry-run` option, which influences how `setupServices` configures the `IFileSystem` (injecting `MemFileSystem`).
        *   When `--dry-run` is used, [`cli.ts`](src/cli.ts:1) (within `setupServices` or related logic) pre-populates the `MemFileSystem` with the content of `*.tool.ts` files from `AppConfig.toolConfigsDir`. This ensures dry runs operate on actual configuration data.
-   **`loadToolConfigs()` Function:** This function, located within [`cli.ts`](src/cli.ts:1) (and also within [`src/modules/config-loader/toolConfigLoader.ts`](src/modules/config-loader/toolConfigLoader.ts:1)), is responsible for loading and parsing all tool configurations. When operating in a dry run, it reads from the pre-populated `MemFileSystem`.
- **CLI Output and Logging:**
    *   The CLI utilizes the `clientLogger` (created per command) for standardized output.
    *   **Verbosity Control:**
        *   `--verbose`: Enables detailed debug messages (`logger.debug()`).
        *   `--quiet`: Suppresses informational (`logger.info()`) and debug (`logger.debug()`) output.
    *   **Standard Output:** Regular informational messages use `logger.info()`.
    *   **Test Environment:** The logger is automatically silenced when `NODE_ENV=test`.
-   **Availability:** The CLI is made available as an executable command (e.g., `mydotfiles`) through the `bin` field in [`package.json`](package.json:1).
